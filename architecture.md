# architecture

## 목적
이 프로젝트는 삼성생명에서 상품 관련 VOC를 응대하는 현장 사용자, 고객센터 민원 담당자, 상품 관련 VOC 담당자가 고객 문의에 대응할 수 있도록 돕는 웹앱이다. 금액 산출 방법이나 근거를 묻는 경우 관련 매뉴얼 경로를 빠르게 확인하고, 현장에서 바로 답변 가능한 건은 즉시 응대할 수 있게 하며, 현장 응대가 어려운 건은 상품 관련 VOC 담당자가 추가 확인할 수 있게 한다. 실제 금액 산출 요청은 자동 제공하지 않은 채 상품팀 담당자 수작업 산출 원칙을 안내한다.

## 설계 원칙
- 완전 자동 응답보다 사람 검토형 보조 구조를 우선한다.
- 검색 가능한 유사 VOC와 답변 초안 자산이 핵심이다.
- 답변 작성, 현장 즉시응대 판단, 추가 확인 연결, 매뉴얼 경로 안내, 금액 산출 문의 구분을 하나의 흐름으로 연결한다.
- 초기에는 반복 VOC 처리에 필요한 최소 기능부터 우선 적용한다.
- 모든 판단과 결과는 추적 가능해야 한다.

## 목표 아키텍처 한 줄 요약
VOC 입력을 받아 `상품군 + 이슈유형`으로 분류하고, 유사 사례를 검색한 뒤, 현장 즉시응대 가능 여부를 판단하고, 금액 산출 방법·근거 문의에는 매뉴얼 경로를 제안하며, 현장 응대가 어려운 건은 상품 관련 VOC 담당자 추가 확인으로 넘기고, 실제 금액 산출 요청에는 자동 제공 불가 및 상품팀 담당자 수작업 산출 원칙을 안내하며, 사람이 최종 검토하는 구조다.

## 상위 구성요소
### 1. VOC 입력 계층
- VOC 원문
- 상품명
- 상품코드
- 기존 VOC data
- 접수 사용자 정보

역할:
- 문의 원문과 기본 정보를 수집한다.
- 이후 분류와 검색에 필요한 최소 구조를 표준화한다.

### 2. 분류 및 검색 계층
- 문의 유형 분류 모듈
- 유사 VOC 검색 모듈
- 현장 즉시응대 가능 여부 판단 모듈

역할:
- VOC를 `상품군 + 이슈유형`으로 분류한다.
- 동일 분류의 기존 VOC를 우선 검색한다.
- 현장이나 고객센터에서 바로 답변 가능한 건인지 판단을 돕는다.

### 3. 근거 및 매뉴얼 계층
- 금액 산출 방법·근거 확인 매뉴얼
- 유사 VOC 답변 자산
- 검색 메타데이터

역할:
- 금액 산출 방법이나 근거를 확인할 수 있는 문서 경로를 제시한다.
- 답변 문장 작성은 기존 VOC 답변 자산을 중심으로 구성한다.
- 매뉴얼은 답변 문장 근거로 직접 쓰지 않는다.

### 4. 초안 생성 계층
- 답변 초안 생성기
- 금액 처리 규칙 적용기

역할:
- 검색 결과를 바탕으로 검토 가능한 초안을 만든다.
- 실제 금액 산출 요청에는 자동산출 불가와 상품팀 담당자 수작업 산출 원칙을 반영한다.

### 5. 검토 계층
- 담당자 검토 화면
- 수정 이력 저장

역할:
- 현장 응대가 어려운 경우 상품 관련 VOC 담당자가 추가 확인한다.
- 사람이 최종 문구를 검토하고 승인한다.
- 작성자 확인 전 등록이나 전송을 하지 않는다.

## 데이터 원칙
- 입력 데이터: 상품명, 상품코드, VOC 내용, 기존 VOC data, 접수 사용자 정보
- 출력 데이터: VOC 답변 초안, 금액 산출 관련 문서/매뉴얼 경로 또는 자동산출 불가 안내
- 참고 자료가 없으면 `근거 없음`으로 표시한다.
- 검색 결과가 0건이면 단정하지 않고 상담사 연결 또는 추가 확인 필요로 표시한다.
- 매뉴얼은 답변 문장 작성 근거로 사용하지 않고, 경로 안내 및 금액 산출 방법·근거 확인용으로만 사용한다.
- 현장 즉시응대가 어려우면 상품 관련 VOC 담당자 추가 확인 필요로 표시한다.
- 실제 금액 산출 요청에는 자동 계산값이나 예시 금액을 제공하지 않는다.
- 실제 금액은 상품팀 담당자가 수작업으로 산출해 제공해야 한다.

## 처리 흐름
1. 사용자가 상품명, 상품코드, VOC 내용을 입력한다.
2. 시스템이 VOC 내용을 `상품군 + 이슈유형` 기준으로 분류한다.
3. 동일 분류의 유사 VOC를 검색한다.
4. 현장이나 고객센터에서 즉시 답변 가능한지 판단한다.
5. 금액 산출 방법·근거 문의이면 관련 매뉴얼 경로를 안내한다.
6. 실제 금액 산출 요청이면 자동 제공 불가와 상품팀 담당자 수작업 산출 원칙을 안내한다.
7. 현장 응대가 어려우면 상품 관련 VOC 담당자가 추가 확인한다.
8. BizRouter LLM이 검색 결과와 입력값을 바탕으로 답변 초안을 작성한다.
9. 사용자가 초안을 검토한다.

## 보안 원칙
- API 키와 비밀값은 서버에서만 사용한다.
- 키나 민감정보는 화면과 로그에 출력하지 않는다.
- 클라이언트에는 필요한 최소 정보만 전달한다.

## 실패 처리
- Supabase 검색 실패 또는 결과 0건: 근거 없음, 상담사 연결 필요
- LLM 실패: `잠시 후 다시`로 안내
- 분류 실패: 보수적으로 유사 검색 범위를 넓히되 추측하지 않음
- 현장 즉시응대 판단 어려움: 상품 관련 VOC 담당자 추가 확인 필요
- 실제 금액 산출 요청: 자동산출 불가, 상품팀 담당자 수작업 산출 안내

## 구현 기준
- 확정되지 않은 내용은 추측하지 않는다.
- 근거가 있는 정보만 답변 초안에 반영한다.
- 현장 즉시응대 가능 여부를 먼저 구분한다.
- 금액 산출 방법이나 근거 문의에는 관련 문서와 매뉴얼 경로만 안내한다.
- 현장 응대가 어려운 건은 상품 관련 VOC 담당자 추가 확인으로 넘긴다.
- 실제 금액 산출 요청에는 자동산출 결과를 제공하지 않는다.
- 실제 금액은 상품팀 담당자가 수작업으로 산출해 제공한다.
- 최종 등록은 사람이 검토한 뒤에만 수행한다.
