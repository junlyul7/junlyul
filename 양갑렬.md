# 상품팀 VOC 응대 지원 AI 에이전트 통합 초안

## 1. 문서 목적
이 문서는 `prd.md`, `customer.md`, `architecture.md`, `AGENTS.md`의 내용을 하나로 합쳐 조원과 함께 검토하기 위한 통합 초안이다.  
목표는 교육 실습용 기획 방향을 한 문서에서 빠르게 공유하고, 이후 최적의 단일 Markdown 문서로 다듬기 쉽게 만드는 것이다.

## 2. 프로젝트 한줄 정의
반복적으로 들어오는 상품 VOC에 대해 `유사 답변 검색 + 근거 문서 추천 + 답변 초안 생성 + 사람 검토` 흐름을 지원하는 교육용 AI 에이전트 기획안이다.

## 3. 배경과 문제 정의

### 현재 문제
- 반복 VOC가 계속 유입되지만 답변 이력이 시스템, FAQ, 개인문서에 분산되어 있다.
- 유사한 문의에 대해 담당자가 매번 다시 답변을 작성한다.
- FAQ, 약관, 상품문서를 다시 찾는 시간이 오래 걸린다.
- 담당자별 경험 차이로 답변 속도와 품질 편차가 발생한다.
- 현장은 자체 해결 가능 여부와 협조 요청 필요 여부를 빠르게 판단하기 어렵다.
- 담당 부서 판단이 애매하면 재배정과 재확인이 발생한다.

### 핵심 문제 한 문장
현장은 무엇을 먼저 찾아봐야 하는지 어렵고, 상품팀은 같은 유형의 답변을 반복 작성하는 구조다.

## 4. 프로젝트 목표
- 반복 상위 VOC를 빠르게 분류하고 재사용 가능한 답변 자산으로 연결한다.
- 과거 답변, FAQ, 약관, 상품문서를 한 흐름에서 탐색할 수 있게 한다.
- 담당자가 최종 답변을 검토하기 전까지 근거 기반 초안을 자동으로 준비한다.
- 현장이 표준 답변만으로 처리 가능한 건을 먼저 해결하도록 지원한다.
- 답변 품질 편차를 줄이고, 건당 처리 시간을 단축한다.

## 5. 대상 사용자

### 직접 사용자
- 상품팀 VOC 답변 담당자
- 현장 VOC 접수 담당자

### 간접 사용자
- 상품기획 실무자
- VOC 운영 관리자
- 신입 직원 및 교육 수강자

## 6. 주요 페르소나

### Persona A. 상품팀 VOC 답변 담당자
- 역할: 상품 관련 VOC 검토 및 답변 작성
- 목표: 빠르고 정확하게 답변하고, 반복 업무를 줄이고 싶다.
- 어려움: 검색 소스가 분산되어 있고, 같은 내용을 반복 작성한다.
- 기대효과: 과거 답변과 근거 문서를 한 번에 보고 초안을 빠르게 만들고 싶다.
- 하루 흐름 예시: 오전에는 협조요청 건을 확인하고, 오후에는 답변 작성과 보정, 문서 확인 업무를 반복한다.

### Persona B. 현장 VOC 접수 담당자
- 역할: 고객 문의 접수 및 1차 판단
- 목표: 자체 해결 가능한 문의는 바로 처리하고 불필요한 협조 요청을 줄이고 싶다.
- 어려움: 표준 답변이 정리되어 있지 않고 담당 부서 판단이 어렵다.
- 기대효과: 문의 유형별 표준 답변과 체크리스트를 먼저 확인하고 싶다.
- 하루 흐름 예시: 문의를 접수할 때마다 자체 해결 가능 여부와 협조 요청 필요 여부를 빠르게 판단해야 한다.

### Persona C. 교육 실습 참여자
- 역할: AI/Codex 기반 업무효율화 흐름을 학습
- 목표: 실제 업무에 가까운 예제로 AI 검색-추천-초안 생성 흐름을 이해하고 싶다.
- 어려움: 기술 개념보다 업무 흐름과 산출물 구조가 먼저 이해되어야 한다.
- 기대효과: 샘플 데이터와 쉬운 단계 설명으로 전체 구조를 따라가고 싶다.

## 7. 고객 pain 우선순위
1. 유사 답변을 다시 찾아 쓰는 시간이 오래 걸린다.
2. 어떤 문서를 근거로 써야 하는지 매번 다시 확인해야 한다.
3. 현장에서 자체 해결 가능한 문의도 상품팀으로 넘어온다.
4. 담당 부서 판단이 애매해 재배정과 재확인이 발생한다.

## 8. 고객이 원하는 가치
- 반복 입력과 반복 작성 감소
- 유사 사례의 빠른 재사용
- 답변 근거의 명확화
- 신입도 따라갈 수 있는 표준화된 처리 흐름
- 현장과 상품팀 간 불필요한 왕복 감소

## 9. 사용자별 기대 차이
- 현장 담당자는 빠른 처리와 바로 쓸 수 있는 표준 답변을 더 중요하게 본다.
- 상품팀 담당자는 정확성, 근거 문서 확인, 최신 표현 사용을 더 중요하게 본다.
- 교육 실습 참여자는 기술 세부 구현보다 전체 업무 흐름의 이해를 더 중요하게 본다.

## 10. 핵심 사용 시나리오
1. 현장 담당자가 VOC 문의를 입력한다.
2. 시스템이 문의 유형을 분류하고 유사 VOC와 과거 답변을 검색한다.
3. 관련 FAQ, 약관, 상품문서, 체크리스트를 함께 제시한다.
4. 처리 가능하면 현장에서 표준 답변을 활용해 종료한다.
5. 추가 검토가 필요하면 상품팀 담당자가 근거를 확인하고 초안을 보정한다.
6. 최종 답변은 사람이 검토 후 VOC 시스템에 입력한다.

## 11. 샘플 VOC 예시
- “이 상품에서 특정 보장이 어떤 조건에서 지급되는지 알려주세요.”
- “가입 후 일정 기간 내 해지하면 환급금이 어떻게 되는지 문의드립니다.”
- “약관상 면책기간이 적용되는 사례인지 확인 부탁드립니다.”
- “기존에 안내된 내용과 다른데 어떤 기준 문서를 봐야 하나요?”
- “이 문의는 현장에서 안내 가능한지, 상품팀 협조가 필요한지 판단 부탁드립니다.”

## 12. MVP 범위

### 포함
- 문의 내용 입력
- 문의 유형 분류
- 유사 VOC / 과거 답변 검색
- 근거 문서 추천
- 체크리스트 제공
- 답변 초안 생성
- 사람이 수정 후 최종 확정하는 흐름

### 제외
- 계산 근거 없는 추론형 답변
- 승인 없는 자동 발송
- 전사 범용 업무 자동화
- 보험료 산출 등 계산 시스템 대체
- 고객정보를 포함한 실제 운영 데이터 사용

### 제외 사유
- 상품 관련 답변은 근거 문서 기반이어야 하며 무근거 생성은 허용하지 않는다.
- 최종 책임과 검토 절차를 유지해야 한다.
- 반복 상위 VOC에 집중하는 편이 실습 효과 검증에 유리하다.
- 본 과제의 목적은 계산이 아니라 답변 지원이다.
- 보안 및 교육 환경 제약이 있다.

## 13. 기능 요구사항

### FR-1. 문의 접수
- 사용자는 VOC 문의 내용을 복사해 입력할 수 있어야 한다.
- 입력 시 상품명, 문의 유형, 접수 채널 등 최소 메타데이터를 함께 기록할 수 있어야 한다.

### FR-2. 문의 분류
- 시스템은 문의 내용을 바탕으로 주요 유형을 제안해야 한다.
- 사용자는 제안된 유형을 수정할 수 있어야 한다.

### FR-3. 유사 사례 검색
- 시스템은 과거 VOC와 표준 답변 후보를 우선순위와 함께 보여줘야 한다.
- 사용자는 검색 결과의 근거 출처를 확인할 수 있어야 한다.

### FR-4. 근거 문서 추천
- 시스템은 FAQ, 약관, 상품안내서, 내부 표준답변 중 관련 자료를 함께 제시해야 한다.
- 추천 자료는 문서명과 사용 목적이 구분되어야 한다.

### FR-5. 체크리스트 제공
- 시스템은 조회 또는 확인이 필요한 항목을 체크리스트 형태로 보여줘야 한다.
- 체크리스트는 답변 전 확인용으로 사용되어야 한다.

### FR-6. 답변 초안 생성
- 시스템은 검색 결과와 근거 문서를 바탕으로 답변 초안을 생성해야 한다.
- 초안에는 근거 문서 기준의 표현을 우선 사용해야 한다.

### FR-7. 사람 검토
- 최종 답변은 사람이 수정, 보정, 승인한 후 사용해야 한다.
- 시스템은 초안과 최종 답변의 차이를 남길 수 있으면 좋다.

## 14. 사용자 스토리
- 현장 담당자로서, 나는 문의를 입력했을 때 먼저 참고할 표준 답변과 체크리스트를 보고 자체 해결 가능 여부를 판단하고 싶다.
- 상품팀 담당자로서, 나는 유사 사례와 근거 문서를 한 번에 확인하고 답변 초안을 빠르게 보정하고 싶다.
- 교육 실습 참여자로서, 나는 입력부터 검색, 근거 확인, 초안 생성, 사람 검토까지의 흐름을 직접 따라가며 이해하고 싶다.

## 15. 비기능 요구사항
- 교육 실습용 샘플 데이터로 동작해야 한다.
- 실제 고객정보, 민감한 내부정보, 비밀값은 사용하지 않는다.
- 초보자도 흐름을 이해할 수 있도록 입력-검색-근거-초안-검토 단계가 명확해야 한다.
- 결과는 복습과 시연에 적합하도록 단순하고 설명 가능해야 한다.

## 16. 성공 지표와 목표 예시

### 성공 지표
- 반복 유형 VOC의 1차 처리 시간 단축
- 표준 답변 재사용률 증가
- 건당 답변 작성 시간 감소
- 담당자 간 답변 편차 감소
- 현장 자체 해결 가능 문의 비중 증가
- 재배정 또는 재확인 요청 건수 감소

### 목표 기준 예시
- 반복 상위 유형의 평균 답변 준비 시간을 실습 기준 30% 이상 단축한다.
- 표준 답변 또는 유사 사례 재사용률을 40% 이상 목표로 둔다.
- 현장 자체 해결 가능 문의 비중을 시범 범위에서 20% 이상 확대한다.
- 재배정 또는 재확인 요청 건수는 감소 추세를 확인한다.

## 17. 측정 방법
- 월 VOC 협조요청 건수 추이
- 반복 유형 협조요청 비중 변화
- 표준 답변 사용률
- 상품팀 건당 평균 답변시간
- 재배정 건수 / 재확인 요청 건수

## 18. AS-IS / TO-BE

### AS-IS
- 답변 이력 분산
- 담당자 경험 의존
- 반복 답변 재작성
- 재배정과 재확인 발생

### TO-BE
- 반복 유형 표준 답변 자산화
- 근거 문서와 체크리스트 동시 제공
- 현장 사전확인 강화
- 상품팀 답변시간 단축과 일관성 확보

## 19. 아키텍처 원칙
- 실제 고객정보 대신 샘플 데이터만 사용한다.
- 계산 시스템이나 운영 시스템을 대체하지 않는다.
- 최종 답변은 사람이 검토한다.
- 설명 가능성과 재현 가능성을 우선한다.
- 복잡한 구조보다 교육용으로 이해하기 쉬운 흐름을 택한다.

## 20. 전체 구조
1. 사용자 입력 계층
2. 문의 분류 및 검색 계층
3. 근거 문서 추천 계층
4. 답변 초안 생성 계층
5. 사람 검토 및 확정 계층

## 21. 논리 구성요소

### 입력 모듈
- VOC 문의 내용과 기본 메타데이터를 입력받는다.

### 분류 모듈
- 문의 내용을 바탕으로 주요 유형을 분류한다.
- 출력: 분류 라벨, 신뢰도, 후속 검색 키워드

### 검색 모듈
- 유사 VOC, 과거 답변, 표준 답변 후보를 탐색한다.
- 조회 대상 예시:
  - VOC 샘플 이력
  - FAQ 문서
  - 표준 답변 모음
  - 담당자 정리 문서

### 근거 문서 추천 모듈
- 약관, 상품문서, 매뉴얼, 체크리스트를 함께 제시한다.

### 초안 생성 모듈
- 검색 결과와 근거 문서를 바탕으로 답변 초안을 생성한다.
- 원칙:
  - 문서 기반 표현 우선
  - 단정적 추론 최소화
  - 사람이 수정하기 쉬운 문장 구조
  - 가능하면 근거 문서명을 함께 표시

### 검토 모듈
- 담당자가 초안을 수정하고 최종 답변을 확정한다.

## 22. 데이터 구조 초안

### VOC 입력 데이터
- `voc_id`
- `received_at`
- `channel`
- `product_name`
- `inquiry_text`
- `requester_role`

### 답변 자산 데이터
- `asset_id`
- `category`
- `question_pattern`
- `standard_answer`
- `source_documents`
- `last_reviewed_at`
- `owner`

### 문서 메타데이터
- `document_id`
- `document_type`
- `title`
- `product_scope`
- `keywords`
- `revision_date`
- `owner`
- `review_status`

## 23. 입력/출력 예시

### 입력 예시
```json
{
  "voc_id": "VOC-001",
  "channel": "VOC",
  "product_name": "샘플 건강보험",
  "inquiry_text": "이 보장이 어떤 경우에 지급되는지 안내 부탁드립니다.",
  "requester_role": "field_staff"
}
```

### 출력 예시
```json
{
  "category": "보장내용 문의",
  "matched_assets": [
    "STD-ANSWER-003",
    "VOC-HISTORY-014"
  ],
  "recommended_documents": [
    "약관 제3조",
    "상품안내서 보장항목 설명"
  ],
  "draft_answer": "관련 약관과 상품안내서를 기준으로 보면 해당 보장은 안내된 지급 조건 충족 시 적용될 수 있습니다. 최종 안내 전 세부 조건 확인이 필요합니다.",
  "needs_human_review": true
}
```

## 24. 처리 흐름
1. 사용자가 VOC 내용을 입력한다.
2. 분류 모듈이 문의 유형을 제안한다.
3. 검색 모듈이 유사 사례와 기존 답변을 찾는다.
4. 근거 문서 추천 모듈이 관련 문서를 제시한다.
5. 초안 생성 모듈이 답변 초안을 만든다.
6. 담당자가 초안을 수정하고 최종 확정한다.
7. 최종 답변과 보정 내용을 답변 자산으로 축적한다.

## 25. 검색 우선순위 규칙
1. 검토 완료된 표준 답변
2. 최근 검토일이 최신인 과거 답변
3. FAQ 또는 공식 상품문서
4. 개인 정리 문서 또는 참고 메모

## 26. 초안 생성 원칙
- 답변은 단정형보다 안내형 문체를 우선 사용한다.
- 근거가 불충분하면 답변을 축약하고 추가 확인이 필요하다고 표시한다.
- 문서 간 표현이 다르면 최신 검토 문서를 우선한다.
- 최종 사용자에게 전달하기 전 내부 검토 필요 여부를 명시한다.

## 27. fallback 규칙
- 유사 사례 없음: “유사 답변 없음”과 함께 수동 검토 안내
- 문서 없음: 초안 생성 대신 관련 확인 항목만 제시
- 분류 충돌: 상위 2개 후보 유형을 함께 제시
- 신뢰도 낮음: 자동 추천보다 사용자의 직접 선택을 우선

## 28. 샘플 구현 구조 제안

### 옵션 A. 단일 로컬 앱
- 간단한 웹 화면 또는 콘솔에서 입력
- 로컬 JSON/Markdown 파일에 샘플 VOC와 문서 저장
- OpenAI API 또는 샘플 규칙 기반 로직으로 분류/초안 생성
- 장점: 실습 준비가 가장 쉽다.

### 옵션 B. 엑셀 + 로컬 스크립트 연계
- 엑셀에서 VOC 입력
- 로컬 스크립트가 검색과 초안 생성 수행
- 결과를 엑셀 또는 Markdown으로 반환
- 장점: 실제 업무 적용 이미지와 가깝다.

### 권장안
- 교육 실습 1차는 `옵션 A`
- 업무 적용 검토 단계에서는 `옵션 B` 확장

## 29. 보안 및 운영 원칙
- 실제 고객정보 입력 금지
- API 키는 환경변수로 분리
- 샘플 데이터만 저장
- 답변 자동 발송 금지
- 최종 승인 없는 외부 전달 금지
- AI는 답변 재사용과 초안 생성을 지원하는 도구다.
- 최종 답변 책임은 사람에게 있다.

## 30. 문서 최신화 흐름
1. 담당자가 최종 답변을 확정한다.
2. 재사용 가치가 있는 답변은 표준 답변 후보로 분류한다.
3. 문서 소유자 또는 운영 담당자가 검토 후 승인 상태를 갱신한다.
4. 최신화된 답변 자산만 우선 검색 대상으로 유지한다.

## 31. 단계별 추진안

### 1단계
- 반복 상위 VOC 유형 선정
- 샘플 데이터 정리
- 표준 답변 후보 자산화

### 1단계 우선 대상 VOC 예시
- 상품 보장 내용 문의
- 보험금 지급 가능 여부 관련 일반 문의
- 약관 해석 요청
- 가입 가능 조건 관련 반복 문의
- 청약 또는 유지 관련 기본 안내 문의

### 2단계
- 검색과 추천 흐름 구현
- 답변 초안 생성 실습
- 현장 사전조회 흐름 시연

### 3단계
- 분류 기준 정비
- 부서별 R&R 반영
- 재배정 감소 효과 검토

## 32. 리스크와 대응
- 검색 품질이 낮을 수 있음
  대응: 표준 답변과 검토 완료 답변을 우선 검색 대상으로 둔다.
- 문서 최신성이 불명확할 수 있음
  대응: 문서별 최종 검토일과 소유자를 메타데이터에 포함한다.
- 현장이 AI 초안을 그대로 신뢰할 수 있음
  대응: 사람 검토 필수 문구와 검토 단계 UI를 명확히 둔다.
- 범위가 빠르게 커질 수 있음
  대응: 1차는 반복 상위 유형만 대상으로 제한한다.

## 33. 도입 저항 포인트
- “AI가 만든 답변이라 바로 믿기 어렵다”는 불안이 있을 수 있다.
- 기존 문서 자산이 정리되어 있지 않으면 추천 품질이 낮게 느껴질 수 있다.
- 현장과 상품팀의 용어 체계가 다르면 분류 기준 정착에 시간이 걸릴 수 있다.
- 초기에 검색 결과가 기대보다 부족하면 활용도가 낮다고 느낄 수 있다.

## 34. 문서 작성 및 협업 원칙
- 교육용 실습과 실제 운영 적용을 구분해서 설명한다.
- 짧은 문장과 실무 표현을 사용한다.
- 추상 설명보다 바로 재사용 가능한 문서 초안, 체크리스트, 샘플 구조를 우선한다.
- 실제 고객정보, 민감자료, 실제 수익성 자료는 사용하지 않고 가짜 샘플 데이터로 대체한다.
- API 키, 인증정보, 내부 시스템 주소 등 비밀값은 문서와 코드에 직접 넣지 않는다.
- 같은 내용을 여러 문서에 반복 복사하지 않고, 한 문서 안에서도 중복을 줄인다.
- “AI가 알아서 판단한다”는 식으로 사람 검토 책임을 흐리지 않는다.

## 36. 조원 리뷰 시 확인하면 좋은 질문
- 1차 MVP 대상 VOC 범위가 너무 넓지 않은가
- 현장과 상품팀 중 어느 사용자에 더 초점을 둘 것인가
- 표준 답변 자산의 최신화 책임자를 어떻게 둘 것인가
- 실습용 구현은 웹 UI가 좋은지, 엑셀 연계 예제가 좋은지
- 검색 정확도와 초안 품질 중 무엇을 먼저 검증할 것인가
